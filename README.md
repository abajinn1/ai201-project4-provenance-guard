# Provenance Guard

Provenance Guard is a backend system for a creative-sharing platform. It accepts a text submission, estimates whether the text appears likely AI-generated, uncertain, or likely human-written, returns a confidence score, displays a reader-facing transparency label, supports creator appeals, applies rate limiting, and records each decision in a structured audit log.

The goal is not to perfectly detect AI writing. The goal is to communicate uncertainty honestly and avoid treating an AI-detection score as absolute proof.

## Architecture Overview

Submission Flow:

    User/Creator
      -> POST /submit
      -> validate text and creator_id
      -> Signal 1: LLM classification
      -> Signal 2: stylistic heuristic score
      -> combine scores into confidence score
      -> choose transparency label
      -> write decision to audit log
      -> return JSON response with content_id, attribution, confidence, label, status, and signal scores

Appeal Flow:

    Creator
      -> POST /appeal
      -> validate content_id and creator_reasoning
      -> find original content decision
      -> update status to under_review
      -> write appeal to audit log
      -> return appeal confirmation

Audit Review Flow:

    Reviewer/Demo
      -> GET /log
      -> return recent structured audit log entries as JSON

## API Endpoints

### POST /submit

Accepts a JSON body with text and creator_id.

Example request:

    {
      "creator_id": "creator-ai-demo",
      "text": "Artificial intelligence represents a transformative paradigm shift in modern society..."
    }

Example response:

    {
      "attribution": "likely_ai",
      "confidence": 0.908,
      "content_id": "7d673ee2",
      "creator_id": "creator-ai-demo",
      "label": "Provenance Guard notice: This post appears likely to have been generated by AI. This label is based on writing-pattern signals and should not be treated as absolute proof.",
      "signals": {
        "heuristic_score": 0.92,
        "llm_score": 0.9
      },
      "status": "classified"
    }

### POST /appeal

Accepts a content_id and creator_reasoning.

Example request:

    {
      "content_id": "039ec162",
      "creator_reasoning": "I wrote this myself as a balanced reflection about remote work. It may sound formal because I was trying to compare both sides clearly, but it was not generated by AI."
    }

Example response:

    {
      "appeal_received": true,
      "content_id": "039ec162",
      "creator_reasoning": "I wrote this myself as a balanced reflection about remote work. It may sound formal because I was trying to compare both sides clearly, but it was not generated by AI.",
      "original_attribution": "uncertain",
      "status": "under_review"
    }

### GET /log

Returns recent audit log entries, including both submissions and appeals.

Example log entry types:

    "event_type": "submission"
    "event_type": "appeal"

## Detection Signals

### Signal 1: LLM-Based Classification

The first signal asks a Groq-hosted LLM to score whether the submitted text appears AI-generated or human-written.

It measures broad semantic and stylistic patterns, including:

- generic formal language
- overly polished structure
- repeated transitions
- lack of concrete personal detail
- language that resembles common AI-generated phrasing

This signal is useful because an LLM can evaluate tone, meaning, specificity, and overall writing feel better than a simple keyword rule.

What it misses: the LLM may be too conservative or may falsely flag polished human writing. Human-written essays, workplace writing, or formal reflections can resemble AI output.

### Signal 2: Stylistic Heuristic Score

The second signal is a Python scoring function that checks observable writing patterns.

It measures:

- AI-like phrases such as "transformative paradigm shift," "it is important to note," "stakeholders," and "ethical implications"
- informal or personal markers such as first-person wording, lived details, and conversational phrasing
- sentence length variation
- short text limitations
- mixed analytical trade-off patterns that may deserve an uncertain label

This signal is useful because it is independent from the LLM. It gives the system a second structured score instead of relying only on the model's judgment.

What it misses: heuristics can be rigid. A human can write formally, and AI can be prompted to include personal-sounding details.

## Confidence Scoring

The system combines the two signal scores into one confidence score from 0.00 to 1.00.

    0.00 = very likely human-written
    0.50 = uncertain or mixed signals
    1.00 = very likely AI-generated

Formula:

    combined_score = (llm_score * 0.60) + (heuristic_score * 0.40)

The LLM signal counts for 60 percent because it can judge semantic and stylistic meaning. The heuristic signal counts for 40 percent because it provides a second, more explainable check.

Thresholds:

    0.75 - 1.00 = likely_ai
    0.45 - 0.74 = uncertain
    0.00 - 0.44 = likely_human

The uncertain range is intentionally wide because false positives can harm creators. A creator should not be labeled likely AI-generated unless the confidence score is high.

## Demo Classification Results

### High-confidence AI-generated example

Input:

    Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications. Furthermore, stakeholders across various sectors must collaborate to ensure responsible deployment.

Observed result:

    attribution: likely_ai
    confidence: 0.908
    llm_score: 0.9
    heuristic_score: 0.92

Why: the text contains multiple generic formal AI-like phrases and a polished structure.

### High-confidence human-written example

Input:

    Ok so I finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put way too much sodium in it and i was thirsty for like three hours after. my friend got the spicy version and said it was better but probably won't go back unless someone drags me there.

Observed result:

    attribution: likely_human
    confidence: 0.078
    llm_score: 0.05
    heuristic_score: 0.12

Why: the text uses informal first-person language, concrete lived details, uneven phrasing, and conversational wording.

### Borderline uncertain example

Input:

    I've been thinking a lot about remote work lately. There are genuine trade-offs: flexibility and no commute on one side, isolation and blurred work-life boundaries on the other. Studies show productivity varies widely by individual and role type.

Observed result:

    attribution: uncertain
    confidence: 0.52
    llm_score: 0.5
    heuristic_score: 0.55

Why: the text has some personal framing but also has a balanced analytical structure and formal trade-off language. The system treats this as mixed evidence rather than forcing a binary label.

## Transparency Labels

The transparency labels are reader-facing and written in plain language. They do not use technical terms like classifier output or logit score.

High-confidence AI label:

"Provenance Guard notice: This post appears likely to have been generated by AI. This label is based on writing-pattern signals and should not be treated as absolute proof."

Uncertain label:

"Provenance Guard notice: This post has mixed writing-pattern signals, so its origin is uncertain. It may be human-written, AI-assisted, or edited from multiple sources."

High-confidence human label:

"Provenance Guard notice: This post appears likely to be human-written based on its writing patterns. This label is an estimate, not a guarantee."

## Appeals Workflow

Creators can appeal a classification by submitting the content_id and creator_reasoning to POST /appeal.

The appeal workflow does not automatically reverse the label. Instead, it:

1. Finds the original content decision.
2. Updates the status to under_review.
3. Stores the creator's reasoning.
4. Writes an appeal event to the audit log.
5. Returns a confirmation response.

In testing, the appeal response showed:

    appeal_received: true
    status: under_review
    original_attribution: uncertain

The audit log also showed the appeal entry beside the original classification entry.

## Rate Limiting

The POST /submit endpoint uses Flask-Limiter.

Chosen limit:

    10 submissions per minute

Reasoning: this is enough for local testing and a short demo, but it prevents unlimited repeated submissions. A real creative-sharing platform would tune this based on normal posting patterns, account age, reputation, and abuse signals.

Observed test result:

    Requests 1-10 returned 200
    Requests 11-12 returned 429

This shows that normal submissions are allowed but repeated rapid submissions are blocked.

## Audit Log

The project uses SQLite for structured audit logging.

Submission entries include:

- timestamp
- event_type
- content_id
- creator_id
- attribution
- confidence
- llm_score
- heuristic_score
- label
- status
- details with text preview and signal reasoning

Appeal entries include:

- timestamp
- event_type
- content_id
- creator_id
- original attribution
- confidence
- signal scores
- label
- status set to under_review
- appeal_reasoning

The audit log is exposed through GET /log for demo and review purposes.

## Known Limitations

### Formal student or workplace writing

The system may misclassify formal human writing as AI-generated because school essays, professional summaries, and policy writing can be polished and structured.

### AI-assisted but human-edited writing

The system cannot reliably separate fully AI-generated writing from human writing that used AI for brainstorming, grammar cleanup, or outlining.

### Short text

Very short submissions may not contain enough writing evidence for a confident decision. These should often be treated as uncertain.

### Creative or poetic writing

Poetry, fiction, and stylized creative writing may use repetition, unusual phrasing, or dramatic structure. The heuristic signal may misread that style.

### Content type most likely to be misclassified

The system would likely misclassify formal academic reflections or professional policy posts because they can contain balanced structure, abstract wording, and polished transitions that overlap with AI-generated writing patterns.

## Spec Reflection

One way the planning spec helped was that it forced the system to use two distinct signals and preserve uncertainty instead of making a simple AI-or-human classifier. The planned thresholds made it easier to create three meaningful transparency labels.

One way the implementation diverged from the plan was the added mixed-signal calibration for borderline analytical writing. During testing, the borderline remote-work example scored too low and was labeled likely_human. I added a mixed analytical trade-off check so that writing with both personal framing and formal analysis lands in the uncertain range.

## Running Locally

Create and activate a virtual environment:

    python -m venv .venv
    source .venv/Scripts/activate

Install dependencies:

    pip install -r requirements.txt

Create a .env file:

    GROQ_API_KEY=your_groq_key_here

Run the app:

    python app.py

Test the homepage:

    curl -s http://localhost:5000/ | python -m json.tool

Test a submission:

    curl -s -X POST http://localhost:5000/submit \
      -H "Content-Type: application/json" \
      -d '{"creator_id":"demo","text":"Artificial intelligence represents a transformative paradigm shift in modern society."}' \
      | python -m json.tool

Test the audit log:

    curl -s http://localhost:5000/log?limit=10 | python -m json.tool
