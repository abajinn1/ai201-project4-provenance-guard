# Provenance Guard Planning

## Project Goal

Provenance Guard is a backend system for a creative-sharing platform. It accepts text from a creator, estimates whether the text appears likely AI-generated, uncertain, or likely human-written, returns a confidence score, displays a reader-facing transparency label, supports creator appeals, applies rate limiting, and records decisions in an audit log.

The goal is not perfect AI detection. The goal is to communicate uncertainty honestly and avoid treating a detection score as absolute proof.

## Detection Signals

### Signal 1: LLM-Based Classification

This signal asks an LLM to classify the submitted text as likely AI-generated, uncertain, or likely human-written. It measures broad semantic and stylistic patterns such as generic phrasing, polished structure, repetitive transitions, lack of personal detail, and language that sounds like a common AI response.

I chose this signal because an LLM can evaluate meaning, tone, specificity, and overall writing feel better than a simple keyword rule.

This signal can miss heavily edited AI writing. It can also falsely flag polished human writing, especially school or professional writing.

### Signal 2: Stylistic Heuristic Score

This signal uses Python rules to inspect writing patterns. It checks sentence length consistency, repeated transition phrases, generic AI-like wording, text length, and whether the writing includes personal or specific details.

I chose this signal because it is independent from the LLM signal. It gives the system a structured score based on observable text features instead of relying only on the model's judgment.

This signal can be too rigid. Some human writing is formal or repetitive, and some AI writing can include specific examples. This signal should not decide the label by itself.

## Confidence Scoring With Uncertainty

The system combines both signals into one confidence score from 0.00 to 1.00.

0.00 means the text appears very likely human-written.
0.50 means the system is uncertain.
1.00 means the text appears very likely AI-generated.

The LLM score counts for 60 percent of the final score because it can judge meaning and tone. The heuristic score counts for 40 percent because it provides a second independent check.

Formula:
combined_score = (llm_score * 0.60) + (heuristic_score * 0.40)

Label thresholds:
- 0.75 to 1.00: likely_ai
- 0.45 to 0.74: uncertain
- 0.00 to 0.44: likely_human

The uncertain range is intentionally wide because a false positive can harm creators. A human creator should not be labeled as AI-generated unless the confidence score is high.

## Transparency Label Design

The label shown to readers will use plain language, not technical jargon.

High-confidence AI label:
"Provenance Guard notice: This post appears likely to have been generated by AI. This label is based on writing-pattern signals and should not be treated as absolute proof."

Uncertain label:
"Provenance Guard notice: This post has mixed writing-pattern signals, so its origin is uncertain. It may be human-written, AI-assisted, or edited from multiple sources."

High-confidence human label:
"Provenance Guard notice: This post appears likely to be human-written based on its writing patterns. This label is an estimate, not a guarantee."

## Appeals Workflow

A creator can submit an appeal if they believe the classification is wrong.

The appeal endpoint requires:
- content_id
- creator_reasoning

When an appeal is submitted, the system will:
1. Find the original content decision by content_id.
2. Mark the content status as under_review.
3. Store the creator's appeal reasoning.
4. Add an appeal entry to the audit log.
5. Return a confirmation response.

The system will not automatically reverse the label. It will mark the content for human review.

## Edge Cases and Known Limitations

### Formal student or workplace writing

A human may write in a polished, structured, or formal way for school or work. The system may incorrectly treat this as AI-generated because polished writing can resemble AI output.

### AI-assisted but human-edited writing

A creator may use AI for brainstorming or grammar cleanup while still writing most of the content themselves. The system may struggle to distinguish fully AI-generated writing from AI-assisted human writing.

### Short text

Very short submissions may not contain enough writing patterns for either signal to evaluate confidently. The system may need to return an uncertain label for short text.

### Creative or poetic writing

Creative writing may use repetition, dramatic phrasing, or unusual structure. A heuristic score may misread those patterns, so the confidence score needs to preserve uncertainty.

## Architecture

Submission Flow:
User/Creator
  -> POST /submit
  -> validate text and creator_id
  -> Signal 1: LLM classification
  -> Signal 2: stylistic heuristic score
  -> combine scores into confidence score
  -> choose transparency label
  -> write decision to audit log
  -> return JSON response with content_id, attribution, confidence, label, status, and signal scores

Appeal Flow:
Creator
  -> POST /appeal
  -> validate content_id and creator_reasoning
  -> find original content decision
  -> update status to under_review
  -> write appeal to audit log
  -> return appeal confirmation

Audit Review Flow:
Reviewer/Demo
  -> GET /log
  -> return recent audit log entries as JSON

## API Endpoints

### POST /submit

Input:
{
  "text": "Submitted creative text goes here.",
  "creator_id": "creator-123"
}

Output:
{
  "content_id": "abc123",
  "creator_id": "creator-123",
  "attribution": "likely_ai",
  "confidence": 0.82,
  "label": "reader-facing transparency label",
  "status": "classified",
  "signals": {
    "llm_score": 0.86,
    "heuristic_score": 0.76
  }
}

### POST /appeal

Input:
{
  "content_id": "abc123",
  "creator_reasoning": "I wrote this myself from my class notes and did not use AI to generate it."
}

Output:
{
  "content_id": "abc123",
  "status": "under_review",
  "appeal_received": true
}

### GET /log

Returns recent structured audit log entries, including submissions and appeals.

## Rate Limiting Plan

The POST /submit endpoint will use Flask-Limiter.

Planned limit:
10 submissions per minute

This is enough for local testing and a small demo, but it prevents unlimited repeated submissions. A real platform would tune this based on normal posting behavior and abuse patterns.

## Audit Log Plan

The audit log will record structured entries for every submission and appeal.

Submission entries will include:
- timestamp
- content_id
- creator_id
- attribution result
- confidence score
- individual signal scores
- transparency label
- status

Appeal entries will include:
- timestamp
- content_id
- creator_id if available
- creator reasoning
- status
- original attribution result if available

The audit log will use SQLite so it is structured and easy to inspect during the demo.

"Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications. Furthermore, stakeholders across various sectors must collaborate to ensure responsible deployment."

Expected result: high AI confidence.

Clearly human-written example:
"Ok so I finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put way too much sodium in it and i was thirsty for like three hours after. my friend got the spicy version and said it was better but probably won't go back unless someone drags me there."

Expected result: lower AI confidence.

Borderline example:
"I've been thinking a lot about remote work lately. There are genuine trade-offs: flexibility and no commute on one side, isolation and blurred work-life boundaries on the other. Studies show productivity varies widely by individual and role type."

Expected result: uncertain or mid-range confidence.
